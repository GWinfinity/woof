# 规则数量对 256 核扩展性的影响分析

## 核心结论

**你的直觉是对的：当前 54 条规则确实太少，导致 256 核扩展性受限。**

但完整的问题是：**固定开销（解析+I/O）占比过高**，而不是单纯的规则数量问题。

## 问题拆解

### 当前瓶颈 (54 条规则)

```
单文件 2.5ms 构成:
┌─────────────────────────────────────┐
│  文件读取 (mmap)     0.5ms  (20%)  │  ← 固定开销
│  tree-sitter 解析    1.0ms  (40%)  │  ← 固定开销
│  54 条规则执行       1.0ms  (40%)  │  ← 可并行
└─────────────────────────────────────┘
```

**问题**: 可并行部分仅占 40%，Amdahl's Law 限制 256 核效率约 30%。

### 目标状态 (400 条规则)

```
单文件 6ms 构成:
┌─────────────────────────────────────┐
│  文件读取 (mmap)     0.5ms   (8%)  │  ← 固定开销
│  tree-sitter 解析    1.0ms  (17%)  │  ← 固定开销
│  400 条规则执行      4.5ms  (75%)  │  ← 可并行
└─────────────────────────────────────┘
```

**改善**: 可并行部分提升到 75%，256 核效率约 38% (+30%)。

## 关键洞察

### 为什么不是单纯增加规则？

1. **Amdahl's Law 硬限制**
   ```
   即使规则无限多，256 核理论最大加速也只有 ~42x
   因为 2% 的串行部分无法并行
   ```

2. **批处理时间悖论**
   ```
   woof 太快导致批处理时间太短
   100K 文件 × 2.5ms = 250 秒总工作
   256 核分摊 = 每核 1 秒
   调度开销 0.2 秒 = 20% 损失
   ```

3. **真正需要两种优化**
   - 减少固定开销（提升效率）
   - 增加规则/分析深度（提升绝对性能）

## 最终回答

> 有没有可能是当前规则过少？

**是的，规则数量是一个重要因素，但不是唯一因素。**

当前 54 条规则导致：
1. ✅ 单文件处理时间太短（2.5ms）
2. ✅ 固定开销占比过高（60%）
3. ✅ 256 核效率受限（29%）

但根本原因是 **固定开销（解析+I/O）无法通过加规则消除**。

**完整解决方案:**
- 短期：优化 I/O 和解析（效率 29% → 45%）
- 中期：增加规则到 200+（绝对性能提升）
- 长期：程序间分析（效率 60%+）

只有**减少固定开销 + 增加计算量**双管齐下，才能实现 256 核的高效利用。
